## 1. Description du traitement (CNIL fascicule 1 §1)

### 1.1 Nature

Le Genesis Engine est un simulateur d'agents synthétiques en
intelligence artificielle / artificial life. Les « agents » de la
simulation sont des structures Python (numpy arrays) sans
correspondance avec une personne physique. Cependant, l'écosystème
d'exécution traite des données personnelles **annexes** :

1. Identité de l'opérateur (auteur, courriel, identifiant Git).
2. Métadonnées de session Claude Code (timestamps, identifiants de
   conversation, chemins de fichiers).
3. Empreinte machine (OS, version Python, chemins absolus contenant
   le nom d'utilisateur OS `micki`).
4. Métadonnées de commits Git et de pull requests (auteur, signataire,
   trailer `Co-Authored-By`).
5. Adresses IP côté Anthropic / GitHub lors des appels API (gérées
   par les sous-traitants).

### 1.2 Portée

| Donnée | Catégorie RGPD | Volume estimé | Sensibilité |
|---|---|---|---|
| Courriel opérateur | Donnée d'identification | 1 enregistrement | Standard |
| UUID conversation Claude | Pseudonyme | ~10/jour | Faible |
| Logs stdout/stderr de smokes | Donnée technique | ~100 KB/run | Faible |
| Manifestes SHA-256 de mondes | Empreinte technique | ~10 KB/monde | Faible |
| Identifiant utilisateur OS | Identifiant indirect | 1 | Faible |
| Empreinte machine (chemins absolus) | Identifiant indirect | omniprésente | Faible |
| Empreinte réseau (IP) côté Anthropic | Donnée d'identification | par appel API | Standard |
| Données spéciales (art. 9) | aucune | 0 | n/a |
| Données d'infraction (art. 10) | aucune | 0 | n/a |
| Données de mineurs | aucune | 0 | n/a |

## 2. Mesures existantes (CNIL fascicule 1 §2)

### 2.1 Mesures conformes au principe de proportionnalité

| Mesure | Conformité |
|---|---|
| Minimisation : seuls les courriels Git, nécessaires à l'attribution, sont collectés | ✅ |
| Limitation des finalités : pas de profilage commercial, pas d'analyse comportementale de tiers | ✅ |
| Exactitude : artefacts Git versionnés ; rectification disponible | ✅ |
| Durée de conservation : durée du projet + 5 ans (norme recherche scientifique) | À documenter |
| Intégrité et confidentialité : SHA-256 manifestes, dépôt privé tant que non publié | ✅ |

### 2.2 Mesures techniques et organisationnelles existantes

- **Authentification** : compte OS Windows 10 protégé par mot de
  passe + chiffrement BitLocker (à confirmer côté opérateur).
- **Versionnement** : Git local + GitHub (privé par défaut).
- **Intégrité des snapshots** : SHA-256 manifest dans
  `world_library.py` (vérification au chargement).
- **Sandbox** : Claude Code en mode permission par défaut (approbation
  par l'opérateur de chaque action de tier-3).
- **Sous-traitance encadrée** : Anthropic Privacy Policy (DPF),
  GitHub Data Processing Addendum.

### 2.3 Mise à jour Wave 11 (objet de la présente révision 1.1)

La Wave 11 (personality-drives-politics, sprint 2026-05-15) introduit
trois mécaniques qui exploitent les traits Big-Five des agents
synthétiques (`ambition`, `extraversion`, `agreeableness`,
`conscientiousness`). **Aucun de ces traits ne se réfère à une
personne physique** : ils sont issus de `prf_rng(seed, …)`. Le
traitement de données personnelles **annexes** (cf. §1.1) reste
identique à la version 1.0 :

- Pas de nouvelle donnée personnelle collectée.
- Pas de nouveau sous-traitant.
- Pas de nouveau pays de stockage.
- Pas de nouvelle finalité.

L'AIPD est donc principalement révisée pour **enrichir le scope
documentaire** et confirmer l'absence d'impact RGPD nouveau.

### 3.4 Gravité et vraisemblance

Niveau de gravité : **2 — limité** (impact maximal = atteinte
réputationnelle de l'opérateur ou perte de données scientifiques).

Niveau de vraisemblance : **2 — limité** (acteur unique, surface
locale, double sous-traitance encadrée).

### 3.5 Nécessité d'un DPO

Critères CNIL Lignes directrices WP243 :
- Traitement à grande échelle de données sensibles ? **Non**.
- Suivi systématique des personnes ? **Non**.
- Activité principale = traitement de données ? **Non** (recherche
  / développement logiciel).

**Conclusion** : la désignation d'un DPO n'est pas obligatoire à ce
stade. Sera réévaluée si le projet ouvre à des utilisateurs tiers ou
intègre des données humaines (cf. §7 Phase 5+).

## 6. Droits des personnes concernées (RGPD chap. III)

Seule personne concernée à ce jour : l'opérateur lui-même. La
procédure ci-dessous s'appliquera si des utilisateurs tiers
s'inscrivent en Phase 5.

| Article | Droit | Procédure |
|---|---|---|
| Art. 13/14 | Information | Présent document publié dans le dépôt + `README.md` §Privacy. |
| Art. 15 | Accès | E-mail à `micka.delcato.rp@gmail.com` ; réponse sous 30 jours. |
| Art. 16 | Rectification | Pull request ou e-mail. |
| Art. 17 | Effacement | Suppression du commit (rewriting Git history) + purge GitHub via `Settings > Security > Account > Personal data`. Logs Anthropic : demande d'effacement via `privacy@anthropic.com` (30 jours). |
| Art. 18 | Limitation | Mise hors-ligne du dépôt (private). |
| Art. 20 | Portabilité | Le dépôt Git est portable par essence. |
| Art. 21 | Opposition | Idem Art. 17. |
| Art. 22 | Décision automatisée | Aucune décision automatisée affectant une personne physique. |

### 6.1 Erasure technique détaillée

1. Identifier tous les commits portant la donnée à effacer :
   `git log --author=<email> --all`.
2. Si donnée présente dans le code (et non simple métadonnée auteur) :
   `git filter-repo --replace-text` ou BFG Repo-Cleaner.
3. `git push --force-with-lease origin --all --tags`.
4. Demander à GitHub la purge du cache (« Removing sensitive data »
   docs).
5. Anthropic : ticket support avec UUID conversation.
