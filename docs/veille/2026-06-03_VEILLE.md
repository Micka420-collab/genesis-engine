## Découvertes

### DÉCOUVERTE_1 — JAX-LaB : Lattice Boltzmann différentiable GPU pour l'hydro/géologie (item roadmap « LBM 2D »)

- **Techno :** **JAX-LaB** (arXiv 2506.17713 ; *Journal of Advances in Modeling Earth Systems*,
  2026, `10.1029/2025MS005313`) est une bibliothèque **Lattice Boltzmann** Python **différentiable**,
  construite sur **JAX** (extension de XLB), pour les **écoulements multiphase / multiphysique en
  géosciences** (milieux poreux hydrologiques et géologiques). Multiphase par méthode
  pseudopotentielle **Shan-Chen** + équation d'état (construction de Maxwell), ratios de densité
  > 10⁸. Exécution **hardware-agnostique** (CPU / multi-GPU / distribué) et **intégration native
  avec le ML** (gradients de bout en bout). À rapprocher de **LEX v1.6.0** (GMD 19/1103/2026), un
  modèle LES **en JAX**, auto-différentiable, pour l'entraînement couplé physique↔deep-learning sur
  GPU/TPU — même tendance « physique différentiable JAX » côté atmosphère.
- **En quoi ça aide Genesis :** la roadmap nomme explicitement, dans « Prochain sprint », **«
  Hydrologie : LBM 2D ou D8 accumulation cross-macro »**. Genesis a déjà le **routage D8 LTI**
  (`discharge_observer`, Wave 53) et l'étage **Exner CPU** proposé le 06-02 ; **JAX-LaB** apporte
  la **branche LBM** alternative/complémentaire, avec en bonus la **différentiabilité** (calibration
  par gradient des paramètres hydro/sédiment contre des cibles d'observation). Distinct de l'Exner
  CPU du 06-02 (schéma volumes finis déterministe) et du « routage rivière différentiable » des
  veilles antérieures (réseau 1D) : ici c'est un **solveur de champ 2D/3D multiphase** sur grille.
- **Module concerné :** hydrologie `sv1d`, `discharge_observer.py`, `geology.py` ; futur kernel
  GPU `native/world-engine` ; banc de calibration hors-ligne (gradients JAX).
- **Effort :** **L** (intégrer un solveur LBM JAX est un chantier ; commencer par un **prototype
  hors-ligne** 2D sur un bassin Genesis, comparé au D8/Exner existant). **M** si on se limite à
  l'usage de JAX-LaB comme **oracle de calibration** sans l'embarquer dans le tick.
- **Risque :** **Moyen-élevé** — **déterminisme** (JAX/GPU : réductions flottantes, `jax.config`
  64-bit, seeds PRNG explicites → garder le D8/Exner CPU comme **source de vérité**, JAX-LaB en
  accélérateur/calibrateur validé contre lui) ; dépendance lourde (JAX) à isoler ; stabilité
  numérique LBM (nombre de Mach, relaxation). Ne **jamais** mettre une sortie apprise dans le
  chemin déterministe de `Simulation.step()` sans validation.
- **Sources :** [JAX-LaB — arXiv 2506.17713](https://arxiv.org/abs/2506.17713) · [JAX-LaB — JAMES 2026 (AGU, 10.1029/2025MS005313)](https://agupubs.onlinelibrary.wiley.com/doi/10.1029/2025MS005313) · [JAX-LaB — code Zenodo](https://zenodo.org/records/15645104) · [LEX v1.6.0 (LES en JAX) — GMD 19/1103/2026](https://gmd.copernicus.org/articles/19/1103/2026/gmd-19-1103-2026.pdf)

### DÉCOUVERTE_2 — Mesurer l'open-endedness : statistiques d'activité évolutive (ToLSim, OEE testée)

- **Techno :** **« A speciation simulation that partly passes open-endedness tests »**
  (arXiv 2603.01701, **2 mars 2026**, de Pinho & Sinapayen) applique les **tests d'open-ended
  evolution (OEE)** — statistiques d'**activité évolutive cumulée** (lignée Bedau-Packard :
  *cumulative*, *mean/median normalized*, *new* evolutionary activity) — à ToLSim. Résultat
  **nuancé et falsifiable** : activité cumulée **non bornée** ✅ mais activité **nouvelle
  persistante = 0** ❌ → « pas (encore) open-ended ». Le cadre méthodologique (composants = gènes
  de l'agent, classes Tokyo type-1 d'OEE) est **réutilisable** indépendamment du substrat. À lire
  avec [On the Open-Endedness of Detecting Open-Endedness (ALife/MIT Press)](https://direct.mit.edu/artl/article/30/3/390/114972/On-the-Open-Endedness-of-Detecting-Open-Endedness).
- **En quoi ça aide Genesis :** c'est une **métrique d'émergence vérifiable**, alignée pile sur
  l'ADN du projet (**ZERO PRE-SCRIPT** + [`FALSIFIABILITY.md`](../../FALSIFIABILITY.md)). Aujourd'hui
  `emergence_metrics.py` mesure des KPIs ; ajouter les **statistiques d'activité évolutive** donne
  un **test objectif** de « la civilisation produit-elle vraiment de la nouveauté soutenue, ou
  plafonne-t-elle ? » sur les runs `terre`/`origins`. Pur **observateur** (lit l'historique des
  gènes/innovations), donc **hors chemin déterministe**, idéal pour une routine de construction.
  Distinct d'ASAL/Leniabreeder (qui *cherchent/génèrent* via foundation models) : ici on **mesure**
  l'open-endedness d'un run existant.
- **Module concerné :** `engine/emergence_metrics.py`, `/api/emergence_metrics`, observateurs
  (`observer_feed`), `FALSIFIABILITY.md` (nouveau critère falsifiable).
- **Effort :** **S→M** (implémenter les compteurs d'activité évolutive Bedau-Packard sur les
  innovations/gènes déjà tracés ; un smoke `p*` qui calcule les courbes sur un run court).
- **Risque :** **Faible** — purement additif et observateur ; veiller à définir proprement le
  « composant » (gène/innovation) et la fenêtre de normalisation pour éviter des artefacts.
- **Sources :** [Speciation & OEE tests — arXiv 2603.01701](https://arxiv.org/abs/2603.01701) · [version HTML](https://arxiv.org/html/2603.01701v1) · [On the Open-Endedness of Detecting Open-Endedness — Artificial Life (MIT Press)](https://direct.mit.edu/artl/article/30/3/390/114972/On-the-Open-Endedness-of-Detecting-Open-Endedness) · [ALife 2026 — programme/ateliers](https://2026.alife.org/workshops-sessions/)

### DÉCOUVERTE_3 — TSL compute shaders (WebGPU) : scaler le rendu d'agents au million côté observation

- **Techno :** **TSL (Three.js Shading Language)** + **WebGPU compute shaders** permettent de
  déplacer **mise à jour de position et instanciation** des particules/agents **entièrement sur
  GPU**, supprimant le transfert CPU↔GPU. Plafond CPU typique ≈ **50 000** particules ; via compute
  WebGPU on atteint **plusieurs millions** (ex. galaxies 750k–1M particules interactives). TSL
  compile vers **WGSL (WebGPU) et GLSL (WebGL)** → un seul code, repli WebGL. Support navigateur
  2026 large (Chrome/Edge/Firefox desktop, **Safari 26** macOS/iOS/visionOS, Chrome Android 12+).
  Guides de **migration Three.js → WebGPU 2026** disponibles.
- **En quoi ça aide Genesis :** **Rendu visuel (82)** + **Observation IA (86)**. L'Earth Console
  affiche déjà des **agents WebGPU** ; un pipeline **TSL compute** permettrait d'**observer des
  populations bien plus denses** en temps réel (overlays agents/flux, replay) sans saturer le CPU,
  et un **repli WebGL** propre. C'est une couche **observation/rendu uniquement** (pas la physique
  déterministe). Distinct du **Gaussian Splatting** (veille 31, *représentation* photoréaliste) :
  ici c'est le **calcul de simulation visuelle de masse** (positions d'agents) sur GPU navigateur.
- **Module concerné :** Earth Console (`earth_console.html`, rendu agents WebGPU), overlays
  d'observation ; aucun impact sur `runtime/` déterministe.
- **Effort :** **M** (porter l'update d'agents/particules de l'Earth Console vers un compute shader
  TSL, garder le repli WebGL).
- **Risque :** **Faible-moyen** — la **physique reste côté serveur** (rendu = miroir) donc pas de
  problème de déterminisme ; risques classiques WebGPU (compat navigateur, perfs mobiles, fallback).
- **Sources :** [Field Guide to TSL & WebGPU — Maxime Heckel](https://blog.maximeheckel.com/posts/field-guide-to-tsl-and-webgpu/) · [Introduction to WebGPU Compute Shaders — Three.js Roadmap](https://threejsroadmap.com/blog/introduction-to-webgpu-compute-shaders) · [Interactive Galaxy with WebGPU Compute Shaders](https://threejsroadmap.com/blog/galaxy-simulation-webgpu-compute-shaders) · [Migrate Three.js to WebGPU (2026) — checklist](https://www.utsubo.com/blog/webgpu-threejs-migration-guide)

### DÉCOUVERTE_4 — Émergence d'un langage-machine entre agents LLM (jeux référentiels, compositionnalité)

- **Techno :** **« Emergence of Machine Language in LLM-based Agent Communication »**
  (OpenReview, **TMLR 02/2026**) montre, via des **jeux référentiels** (locuteur→message dans un
  alphabet artificiel de 16 syllabes ; auditeur identifie la cible parmi des distracteurs), que
  **deux agents LLM développent un langage partagé** (541 objets appris en ~4 tours, ≤3 essais),
  langage présentant **compositionnalité, généralisation, morphèmes, polysémie** — propriétés du
  langage humain — **sans protocole prédéfini**. À relier au cadre théorique **« Generative
  emergent communication / collective world model »** (Advanced Robotics, 2026).
- **En quoi ça aide Genesis :** **Sociétés / agents (76)**. Genesis a déjà **lexique + memetic +
  `/api/audio`** ; ce travail donne un **protocole expérimental** (referential game) et des
  **métriques** (compositionnalité, généralisation) pour faire **émerger un lexique non scripté**
  et le **mesurer**. Pertinent surtout pour **Phase 5 (LLM tier-2, ⏳)** : à classer comme
  **inspiration de design + métrique**, pas dépendance de production immédiate. Distinct de la
  mémoire long-horizon d'agents LLM (veille antérieure) et du RWML : ici c'est la **formation de
  langage** elle-même.
- **Module concerné :** cognition agents (lexique/vocalisations), `emergence_metrics`
  (compositionnalité), futur tier-2 LLM (Phase 5).
- **Effort :** **M** (protocole referential-game offline pour mesurer la compositionnalité du
  lexique émergent actuel) → **L** si couplage à un tier-2 LLM.
- **Risque :** **Moyen** — **déterminisme/coût** si LLM dans la boucle (inférence sandboxée,
  cache, seeds) ; risque de **script déguisé** (garder l'émergence du sens, pas un dictionnaire
  câblé) ; rester en R&D tant que Phase 5 n'est pas débloquée.
- **Sources :** [Emergence of Machine Language in LLM-based Agent Communication — OpenReview (TMLR 02/2026)](https://openreview.net/forum?id=zy06mHNoO2) · [PDF](https://openreview.net/pdf/dd385254607d317329de7f1ab96728b480363cb4.pdf) · [Generative emergent communication (collective world model) — Advanced Robotics 2026](https://www.tandfonline.com/doi/full/10.1080/01691864.2026.2661958)

## CVE / sécurité (stack & supply-chain) — **veille du jour**

> **Nouveauté critique du jour : CVE-2026-48710 « BadHost ».** Bypass d'authentification dans
> **Starlette** (le socle de **FastAPI**, vLLM, serveurs MCP) via un **en-tête `Host` malformé**.
> **Directement pertinent pour Genesis** : l'**Earth Console** est un serveur **SSE FastAPI**
> (`/api/events/stream`, `make earth-console` sur :8090) — exactement la classe d'endpoints
> (`/sse`, `/messages`) citée comme exploitable.

| Réf. | Sujet | Pertinence Genesis |
|------|-------|--------------------|
| **CVE-2026-48710 — « BadHost » (Starlette/FastAPI)** | Un caractère invalide RFC 9112 §3.2 (`/`, `?`, `#`) dans l'en-tête `Host` décale les bornes path/query → `request.url.path` diffère du chemin réellement routé, **contournant les contrôles d'accès** des middlewares. Touche serveurs FastAPI, proxies OpenAI-compatibles, **serveurs MCP** (`/mcp`, `/sse`, `/messages`). | **Très pertinent** : Earth Console = **SSE FastAPI**. **Reco : monter à `starlette >= 1.0.1` ; dans tout middleware de sécurité, remplacer `request.url.path` par `request.scope["path"]` ; ne pas exposer la console hors `127.0.0.1` sans garde.** |
| **TrapDoor — supply-chain + empoisonnement IA (rappel 06-01/06-02)** | 34 paquets / 384 versions sur **npm+PyPI+crates.io** (vol SSH/AWS/GitHub/wallets) ; **`build.rs`** malveillant côté Rust (XOR `cargo-build-helper-2026` → GitHub Gists) ; **plante des `CLAUDE.md`/`.cursorrules`** piégés (Unicode largeur nulle) pour détourner les assistants IA. Première activité **22 mai 2026**. | **crates.io alimente `native/world-engine`** + **routines IA nocturnes**. `cargo audit` + **`cargo-deny`/`cargo-vet`**, **épingler** les versions, **lint anti-empoisonnement** sur `CLAUDE.md`/`AGENTS.md`/`.cursorrules`/`.claude/` (cf. reco 06-02). |
| **Pistes ML (DÉC. 1 & 4)** | Dépendances lourdes (JAX) et inférence LLM potentielle | Inférence **sandboxée**, sorties = **observation/prior** jamais source de vérité ; épingler+hash JAX et toolchain ; pas de `build.rs`/postinstall non audité. |

**Recommandations cybersécurité :**
1. **BadHost (urgent)** : vérifier la version de Starlette (`pip show starlette`), **monter ≥ 1.0.1** ;
   auditer le middleware de l'Earth Console — utiliser `request.scope["path"]` pour toute décision
   de sécurité ; **garder la console liée à `127.0.0.1`** (pas d'exposition réseau) ; scanner via
   [badhost.org](https://badhost.org/) si un endpoint est joignable.
2. **TrapDoor / crates.io** : `cargo audit` + **`cargo-deny`** (advisories + sources + licences) en
   CI ; **épingler** les versions ; auditer toute nouvelle dépendance Rust (et **`build.rs`**) avant
   bump.
3. **Anti-empoisonnement IA** : lint CI détectant les **caractères Unicode invisibles / largeur
   nulle** dans `CLAUDE.md`, `AGENTS.md`, `.cursorrules`, `.claude/`, `.github/` ; **revue humaine**
   des diffs touchant ces fichiers de contexte agent (routines nocturnes).
4. **CI / wheels** (rappel) : runners isolés, **trusted publishing OIDC**, pinning+hash de
   l'outillage (maturin/uv/JAX), rotation des secrets ; `pip-audit` + `cargo audit` maintenus.
5. **Pistes ML (DÉC. 1-4)** : inférence **sandboxée**, sorties traitées comme **observation/prior**
   et **jamais** comme source de vérité de la simulation déterministe (`Simulation.step()`).
