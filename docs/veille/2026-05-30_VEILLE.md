## Découvertes

### DÉCOUVERTE_1 — Routage de rivière différentiable sur GPU (LTI = convolution block-sparse)

- **Techno :** *Differentiable River Routing for End-to-End Learning of Hydrological
  Processes* (Hascoet, Pellet, Oishi et al.). Idée clé : les schémas de routage
  **linéaires invariants dans le temps (LTI)** sont **mathématiquement équivalents** à des
  couches de convolution **block-sparse** — la dimension spatiale des chenaux ↔ la
  dimension « canaux » d'une conv. D'où **accélération GPU + différentiabilité automatique**,
  et reproduction du run GEOGloWS global **sur un seul GPU**.
- **En quoi ça aide Genesis :** c'est le levier le plus direct sur la dimension Hydrologie
  (70 %) — gaps « bassins versants dynamiques » et « D8 accumulation cross-macro ». Le
  routage des débits sur le réseau hydro (déjà quantifié par `watershed_observer` :
  Strahler / Horton / drainage density) pourrait être propagé via un opérateur
  conv block-sparse rapide et déterministe, sans dépendre d'un schéma de routage externe.
- **Module concerné :** `watershed_observer.py`, hydrologie runtime (`hydrology_mode` sv1d / D8),
  éventuellement crate `hydrology` côté Rust.
- **Effort :** **M-L** (portage de l'opérateur LTI→conv en numpy/torch, branchement sur le
  réseau de drainage existant, validation vs `sv1d`).
- **Risque :** **Moyen** — parité déterministe CPU↔GPU à garantir pour les smokes ; calage
  des paramètres de routage ; dépendance torch/GPU à isoler hors hot path déterministe.
- **Sources :** [JGR-MLC 2026 (Wiley)](https://agupubs.onlinelibrary.wiley.com/doi/10.1029/2025JH000760?af=R) · [ESS Open Archive (preprint, 27 mai 2025)](https://essopenarchive.org/doi/full/10.22541/essoar.174835108.87664030/v1) · [PDF preprint](https://d197for5662m48.cloudfront.net/documents/publicationstatus/260455/preprint_pdf/58d668932daa70d67439d8f4d8d08e41.pdf) · [HESS 2025 — ensembles différentiables streamflow](https://hess.copernicus.org/articles/29/6829/2025/)

### DÉCOUVERTE_2 — AgentSociety 2 : plateforme open-source d'agents LLM à grande échelle

- **Techno :** *AgentSociety* (Tsinghua FIB Lab, arXiv 2502.08691) — simulateur social
  open-source d'agents génératifs pilotés par LLM : couches Model / Agent / LLM, moteur
  parallèle **Ray**, jusqu'à **~30 000 agents** plus vite que le temps réel, **replay
  d'expériences**, multi-LLM (OpenAI, **Qwen, Deepseek** — donc auto-hébergeable). AgentSociety 2
  se présente comme « LLM-native » pour la recherche en sciences sociales.
- **En quoi ça aide Genesis :** architecture de référence pour l'**unlock LLM tier-2 (Phase 5)**
  sans réinventer l'orchestration. Le découplage couche Agent / couche LLM + replay est
  compatible avec notre philosophie ZERO PRE-SCRIPT et nos `emergence_metrics` ; le backend
  Ray éclaire la montée en charge multi-agents. À utiliser comme **source d'inspiration
  architecturale**, pas comme dépendance importée telle quelle.
- **Module concerné :** agents / cognition (`runtime/`), `civilization_pipeline.py`, plan
  Phase 5 (`../Genesis_Engine_2026-05-23_Phase5_Unlock_Brief.md`).
- **Effort :** **L** (intégration LLM tier-2, harnais d'inférence local Qwen/Deepseek,
  garde-fous coût/repro).
- **Risque :** **Moyen-élevé** — non-déterminisme LLM (à confiner hors cœur déterministe),
  coût d'inférence, validation difficile (cf. revue critique ci-dessous). **Cyber :** si LLM
  auto-hébergé, isoler l'inférence (pas d'exécution d'outil arbitraire par l'agent ;
  sandbox réseau).
- **Sources :** [arXiv 2502.08691](https://arxiv.org/abs/2502.08691) · [GitHub tsinghua-fib-lab/AgentSociety](https://github.com/tsinghua-fib-lab/agentsociety) · [MarkTechPost — présentation](https://www.marktechpost.com/2025/07/31/agentsociety-an-open-source-ai-framework-for-simulating-large-scale-societal-interactions-with-llm-agents/) · [Validation = défi central (revue, PMC 2025)](https://pmc.ncbi.nlm.nih.gov/articles/PMC12627210/)

### DÉCOUVERTE_3 — World models désormais open-source (alternatives intégrables à Genie 3)

- **Techno :** trois sorties **début 2026** lèvent le verrou « propriétaire » noté le 29 sur
  Genie 3 : **LingBot-World** (Robbyant, 28 janv. 2026, **Apache-2.0**, temps réel < 1 s de
  latence à 16 fps, horizon « minute » avec long-term memory, poids sur Hugging Face) ;
  **NVIDIA DreamDojo** (world model robotique, distillation *Self Forcing* ~10,8 fps) ;
  **AMD Micro-World** (modèles interactifs contrôlés par actions, train + inférence).
- **En quoi ça aide Genesis :** notre `World_Models_Adapter_Spec` visait Genie 3 mais le
  jugeait non intégrable (propriétaire, US-only). **LingBot-World en Apache-2.0** redevient
  une cible réaliste pour un adaptateur read-only / preview observationnel côté Earth Console
  (Observation IA 86 %, Rendu 82 %) — génération de « prévisions » visuelles à partir d'un
  état monde, sans toucher le cœur déterministe.
- **Module concerné :** adaptateur world-model (`docs/...World_Models_Adapter_Spec`),
  observation / Earth Console (read-only).
- **Effort :** **L** (portage modèle, GPU d'inférence, mapping état Genesis → conditionnement).
- **Risque :** **Moyen-élevé** — modèles vidéo lourds (VRAM/latence), sortie non
  déterministe (à cadrer comme couche d'observation, jamais source de vérité de sim) ;
  vérifier la licence des **poids** (Apache-2.0 annoncée) en plus du code.
- **Sources :** [arXiv 2601.20540 — Advancing Open-source World Models](https://arxiv.org/abs/2601.20540) · [GitHub Robbyant/lingbot-world](https://github.com/Robbyant/lingbot-world) · [Hugging Face — lingbot-world-base-act](https://huggingface.co/robbyant/lingbot-world-base-act) · [NVIDIA DreamDojo (MarkTechPost)](https://www.marktechpost.com/2026/02/20/nvidia-releases-dreamdojo-an-open-source-robot-world-model-trained-on-44711-hours-of-real-world-human-video-data/) · [AMD Micro-World (ROCm Blogs)](https://rocm.blogs.amd.com/artificial-intelligence/micro-world/README.html)

### DÉCOUVERTE_4 — Neural Cellular Automata compacts (multi-textures, mesh, terrain 3D depuis CA 2D)

- **Techno :** avancées 2025-2026 sur les **NCA** : un **automate compact unique** capable
  de faire évoluer **plusieurs textures** (Nature Scientific Reports 2025), **Mesh-NCA**
  (textures directement sur maillages, ACM TOG), et « Generating 3D Terrain with 2D Cellular
  Automata » (IEEE CoG 2025).
- **En quoi ça aide Genesis :** Genesis utilise déjà des NCA (settlements). Un NCA
  multi-textures/mesh apporterait une **texturation procédurale légère et déterministe** des
  biomes/relief (appui Rendu 82 % et procgen géologie 64 %), apprise hors-ligne puis
  rejouée à coût quasi nul à l'inférence (local, pas de GIL, pas de LLM).
- **Module concerné :** procgen terrain/biome (crate `terrain` / `biome`), couche rendu /
  texturation Earth Console.
- **Effort :** **M** (entraînement NCA offline, export poids compacts, intégration texturation).
- **Risque :** **Moyen** — coût d'entraînement initial, contrôle artistique limité,
  intégration GPU/CPU à valider sur renders de compliance.
- **Sources :** [Multi-texture NCA (Nature Sci. Rep. 2025)](https://www.nature.com/articles/s41598-025-23997-7) · [Mesh-NCA (ACM TOG)](https://dl.acm.org/doi/10.1145/3658127) · [Curated world/CA models (GitHub)](https://github.com/nik-55/world-models)

## CVE / sécurité (stack & supply-chain) — **action recommandée ce jour**

> ⚠️ **Impact direct sur le build Genesis** : une CVE PyO3 abi3 touche **notre configuration**.

| Réf. | Sujet | Pertinence Genesis |
|------|-------|--------------------|
| **PyO3 — type confusion abi3 (divulguée 18 fév. 2026)** | Confusion de type à l'accès de données de **sous-classes de sous-classes** de types natifs, avec la feature **`abi3` ciblant Python ≥ 3.12** | **DIRECT** : nos bindings sont buildés `abi3-py39`. Si exécutés/buildés contre CPython ≥ 3.12, l'exposition existe. **Reco : bump `pyo3` vers une version corrigée et vérifier l'usage de sous-classes de types natifs.** |
| **CVE-2026-33056 (Cargo)** | Une crate malveillante peut **changer les permissions de répertoires arbitraires** lors de l'extraction par Cargo au build | **CI / supply-chain** : pertinent pour le sprint « CI maturin wheel ». **Reco : mettre à jour la toolchain Rust/Cargo, builder en runner isolé.** |
| **RUSTSEC-2026-0060** | `tokio-timer` **non maintenu** (écosystème tokio 0.1) | Non utilisé (Genesis = `tokio 1.40`). Vérifier l'absence de transitive legacy via `cargo audit`. |

**Recommandations cybersécurité :**
1. **Bumper `pyo3`** vers une release corrigeant la type-confusion abi3, et re-tester les smokes du pont (`p82`).
2. **Mettre à jour Cargo/rustup** et builder les wheels maturin en **runner CI isolé** (sandbox), épingler les versions de l'outillage (maturin/uv) + vérifier les hash.
3. Maintenir **`cargo audit` + `pip-audit`** en CI (déjà recommandé le 29).
4. Pour les pistes LLM/world-model (DÉC. 2 et 3) : **inférence sandboxée**, pas d'exécution d'outil arbitraire par les agents, sorties traitées comme observation et **jamais** comme source de vérité de la simulation déterministe.

**Sources :** [RustSec Advisories](https://rustsec.org/advisories/) · [PyO3 advisories](https://rustsec.org/packages/pyo3.html) · [CVE-2026-33056 — Rust Blog](https://blog.rust-lang.org/2026/03/21/cve-2026-33056/) · [RUSTSEC-2026-0060](https://db.gcve.eu/vuln/rustsec-2026-0060)
