## Découvertes

### DÉCOUVERTE_1 — Morphodynamique Shallow Water–Exner : noyau physique pour l'érosion dynamique (sedExnerFoam 2412, composants Landlab)

- **Techno :** la voie standard du transport sédimentaire couple les équations **Saint-Venant
  (shallow water)** à l'**équation d'Exner** (conservation de la masse de sédiment → évolution
  du lit) : c'est le modèle **Shallow Water–Exner**. Nouveautés citables 2025-26 :
  **`sedExnerFoam` 2412** (GMD 19/2299/2026) — modèle 3D Exner couplant hydrodynamique
  haute-résolution + concentration de sédiment en suspension + module morphologique ;
  **RiverBedDynamics v1.0** (GMD 18/3427/2025) et **SPACE 1.0**, composants **Landlab** Python
  pour le transport 2D, l'érosion du socle rocheux et l'évolution de paysage (LEM) ; schémas
  **volumes finis semi-implicites** pour Exner (arXiv 2303.03801) et **approche par moments**
  hydro-morphodynamique couplée (arXiv 2505.22278). Une implémentation **GPU** auto-cohérente
  shallow-water + sédiment existe (arXiv 2103.15196).
- **En quoi ça aide Genesis :** c'est **exactement le prochain palier de la roadmap** —
  « reste **érosion GPU dynamique (shallow-water / sédiment transport)** comme prochain palier »
  (géologie 68, hydrologie 72 → 75 %+). Genesis a déjà le **routage de débit LTI D8**
  (`discharge_observer`, Wave 53, conservation de masse exacte) et la **quantification réseau**
  (Strahler/Horton, `watershed_observer`) : un étage **Exner** ajoute le **lit mobile** (érosion/
  dépôt) au-dessus du débit déjà calculé, fermant la boucle eau→sédiment→relief. Distinct de la
  veille 29 (« érosion GPU » abstraite) : ici, des **modèles validés et publiés** (Landlab
  Python testable tout de suite côté CPU déterministe, sedExnerFoam comme référence physique).
- **Module concerné :** `geology.py`, `discharge_observer.py`, `watershed_observer.py`,
  hydrologie sv1d ; futur kernel `native/world-engine` (crate `gpu`) pour la version GPU.
- **Effort :** **M** (prototype Exner CPU couplé au débit D8 existant, validé par smoke) →
  **L** (portage GPU shallow-water + Exner, dans la lignée `bevy_gpu_compute` du 06-01).
- **Risque :** **Moyen** — **déterminisme** (réductions flottantes GPU → d'abord noyau CPU
  déterministe de référence, GPU validé contre lui) ; stabilité numérique du couplage Exner
  (CFL, schémas semi-implicites) ; coût compute. Garder la physique CPU comme **source de
  vérité** et la version GPU comme accélérateur validé.
- **Sources :** [sedExnerFoam 2412 — GMD 19/2299/2026](https://gmd.copernicus.org/articles/19/2299/2026/) · [RiverBedDynamics v1.0 (Landlab) — GMD 18/3427/2025](https://gmd.copernicus.org/articles/18/3427/2025/) · [Exner volumes finis semi-implicites — arXiv 2303.03801](https://arxiv.org/pdf/2303.03801) · [Couplage hydro-morphodynamique par moments — arXiv 2505.22278](https://arxiv.org/pdf/2505.22278) · [GPU shallow-water + sédiment — arXiv 2103.15196](https://arxiv.org/pdf/2103.15196)

### DÉCOUVERTE_2 — Espace d'action compositionnel : Conditional Action Trees + bibliothèques de compétences (gap « ActionKind enum »)

- **Techno :** au lieu d'un **espace d'action discret figé**, des approches **compositionnelles /
  hiérarchiques** structurent l'action : **Conditional Action Trees** (arXiv 2104.07294)
  généralisent les espaces d'action discrets en arbres conditionnels (actions valides
  dépendantes de l'état) compatibles A2C/SAC ; les **bibliothèques de compétences** (skill
  libraries) fournissent des **primitives réutilisables** composables ; **CrossAgent**
  (arXiv 2512.09706, déc. 2025) entraîne **un seul modèle** à maîtriser **plusieurs espaces
  d'action hétérogènes** (API haut niveau ↔ primitives bas niveau) et à **choisir
  dynamiquement** le bon espace à chaque pas. Thème confirmé comme axe de recherche 2026
  (AAMAS 2026, émergence de spécialisation).
- **En quoi ça aide Genesis :** **Sociétés / agents (76 %)**, gap explicite « `ActionKind`
  encore enum » de la roadmap. Remplacer l'enum par un **arbre d'actions conditionnel** (ou une
  bibliothèque de compétences composables) permettrait des comportements plus riches
  **(construire, troquer, combiner)** sans script — composition **émergente** plutôt
  qu'énumération câblée. Aligné **ZERO PRE-SCRIPT** si les compétences se composent par
  apprentissage/utilité, pas par recettes. Complète `NEAT + latent_action` déjà en place.
  Distinct du **RWML** (06-01, *prédiction* de dynamique) et de l'**auto-télique** (veille 31,
  *génération de buts*) : ici on travaille la **représentation de l'action** elle-même.
- **Module concerné :** `runtime/` (agents, `ActionKind`/`latent_action`, NEAT),
  `emergence_metrics` (mesurer la diversité d'actions composées).
- **Effort :** **M** (refactor `ActionKind` enum → structure compositionnelle rétro-compatible ;
  validité conditionnelle par état ; smoke émergence).
- **Risque :** **Moyen** — complexité accrue (sur-ingénierie possible vs règle d'utilité
  actuelle) ; **déterminisme** si apprentissage de compétences (seed/cache) ; veiller à ne pas
  réintroduire de script déguisé dans la composition.
- **Sources :** [Conditional Action Trees — arXiv 2104.07294](https://arxiv.org/pdf/2104.07294) · [CrossAgent (multi action-space) — arXiv 2512.09706](https://arxiv.org/pdf/2512.09706) · [Compositional RL & skill libraries (logique temporelle) — PMC8326833](https://www.ncbi.nlm.nih.gov/pmc/articles/PMC8326833/) · [AAMAS 2026 (émergence multi-agents)](https://mas-space.github.io/aamas2026ws/)

### DÉCOUVERTE_3 — Agents spatiaux + world models : perception→action ancrée dans l'espace

- **Techno :** **« From Perception to Action: Spatial AI Agents and World Models »**
  (arXiv 2602.01644, fév. 2026) articule des **agents à raisonnement spatial** couplés à des
  **world models** pour la boucle **perception → action** ancrée géométriquement dans
  l'environnement. À distinguer nettement : (a) des **world models vidéo génératifs** (Genie 3,
  veille 30) qui *rendent* des mondes ; (b) du **RWML** (06-01) qui apprend une *dynamique*
  action-conditionnée interne. Ici l'accent est la **localisation/structure spatiale** de la
  perception et de l'action.
- **En quoi ça aide Genesis :** **Observation IA (86 %)** + **Sociétés / agents (76 %)**.
  Genesis perçoit déjà localement (perceive→decide→act) sur une grille ; un cadre d'**ancrage
  spatial** explicite pourrait enrichir la **mémoire d'observation** (`observer_feed`) et la
  cohérence spatiale des décisions multi-générations (territoires, routes émergentes). Usage
  **inspiration de design / observation**, pas dépendance de production.
- **Module concerné :** `observer_feed`, cognition agents (perception/mémoire spatiale),
  Earth Console (overlays d'observation).
- **Effort :** **M** (intégration côté perception/observation ; surtout R&D conceptuelle).
- **Risque :** **Faible-moyen** — recherche récente (à valider sur un cas Genesis concret) ;
  ne pas confondre rendu génératif et structure spatiale ; garder hors du chemin déterministe
  si un modèle appris est introduit.
- **Sources :** [Spatial AI Agents & World Models — arXiv 2602.01644](https://arxiv.org/pdf/2602.01644) · [Universal & Interpretable World Models for Open-ended Agents — arXiv 2409.18676](https://arxiv.org/pdf/2409.18676)

## CVE / sécurité (stack & supply-chain) — **veille du jour**

> **Mise à jour critique** de la note du 06-01 sur **TrapDoor** : au-delà du vol de credentials
> sur **npm / PyPI / crates.io**, l'analyse détaillée révèle un **vecteur d'empoisonnement des
> assistants IA** — directement pertinent pour Genesis, dont les **routines nocturnes sont
> pilotées par IA** et dont le dépôt contient des fichiers de contexte agent (`.claude/`, `AGENTS.md`).

| Réf. | Sujet | Pertinence Genesis |
|------|-------|--------------------|
| **TrapDoor — empoisonnement IA (mai 2026)** | Les paquets plantent des fichiers **`CLAUDE.md` et `.cursorrules`** contenant des **instructions cachées en Unicode largeur nulle**, parsées par Claude Code / Cursor pour injecter du contexte projet **à l'insu du développeur** ; persistance via cron/systemd/**git hooks**, `build.rs` malveillant côté Rust (Sui/Move) | **Très pertinent** : Genesis = repo piloté par routines IA + crates Rust (`native/world-engine`). **Reco : scanner `CLAUDE.md`/`AGENTS.md`/`.cursorrules`/`.claude/` pour caractères de contrôle Unicode ; revue humaine de tout fichier de contexte agent ; auditer les git hooks.** |
| **TrapDoor — supply-chain (rappel 06-01)** | **34 paquets / 384 versions** sur npm+PyPI+**crates.io**, vol SSH/AWS/GitHub/wallets ; détection Socket médiane ~5 min 27 s | **crates.io alimente le workspace Rust.** `cargo audit` + `cargo-deny`/`cargo-vet`, **épingler** les versions, vérifier l'origine avant tout bump (ex. `bevy_gpu_compute`, montée Bevy 0.18). |
| **Sécurité multi-agents (arXiv 2505.02077)** | Défis ouverts de **sécurité des systèmes d'agents en interaction** (injection, miscoordination) | Cadre utile si Genesis introduit des agents LLM tier-2 (Phase 5) : **isoler/valider** les entrées inter-agents. |

**Recommandations cybersécurité :**
1. **Anti-empoisonnement IA (nouveau)** : ajouter un **lint CI** détectant les **caractères
   Unicode invisibles / largeur nulle** dans `CLAUDE.md`, `AGENTS.md`, `.cursorrules`, `.claude/`,
   `.github/` ; **revue humaine obligatoire** des diffs touchant ces fichiers de contexte agent.
2. **crates.io / TrapDoor** : `cargo audit` + **`cargo-deny`** (advisories + sources + licences)
   en CI ; **épingler** les versions ; auditer toute nouvelle dépendance Rust avant montée.
3. **Git hooks & build scripts** : inspecter `build.rs` des deps Rust et les hooks Git
   (persistance TrapDoor) ; **pas de postinstall/`build.rs` non audité**.
4. **CI / wheels** (rappel) : runners isolés, **trusted publishing OIDC**, pinning+hash de
   l'outillage (maturin/uv), rotation des secrets longue durée ; `pip-audit` + `cargo audit`
   maintenus.
5. **Pistes ML (DÉC. 1-3)** : inférence **sandboxée**, sorties traitées comme **observation/prior**
   et **jamais** comme source de vérité de la simulation déterministe (`Simulation.step()`).
