## TL;DR

`engine/cognition.py` — deux changements groupés dans une seule passe :

1. **`r_chunks` resserré** : `ceil(radius / CHUNK_SIDE_M) + 1` → `ceil(radius / CHUNK_SIDE_M)`. À `PERCEPTION_RADIUS_M = 60 m` et `CHUNK_SIDE_M = 32 m`, la fenêtre passe de 7×7 = 49 chunks à 5×5 = 25 chunks (**-49 %** sur le coût de la boucle bbox-prefilter elle-même). Le +1 était une garde redondante : la preuve géométrique tient pour tout `radius < 2·CHUNK_SIDE_M` (le bord intérieur d'un chunk à offset r est à au moins `(r-1)·CHUNK_SIDE_M`, donc r=3 dépasse 60 m d'office).

2. **Cache d'indices clairsemés par (chunk, tick)** : nouvelle fonction `_chunk_resource_indices(chunk, tick)` qui retourne `((water_y, water_x), (food_y, food_x), (shelter_y, shelter_x))` — les `np.nonzero` des trois prédicats de ressource. Cache attaché à l'instance `chunk._scan_idx` + `chunk._scan_idx_tick`. Recalcule au maximum une fois par tick par chunk. `_scan_chunk` itère désormais uniquement sur les cellules clairsemées (souvent 0–500 cellules) au lieu d'allouer une bool-mask (4096 cellules × 3 ressources × 2 ops chacune) à chaque appel.

3. **Threading du `tick`** : `perceive(...)` accepte maintenant `tick=None`, et `Simulation.step` passe `tick=self.tick`. Si `tick=None` le cache est désactivé (fallback dégradé, comportement = ancienne version sauf la fenêtre `r_chunks`).

## Justification géométrique du `r_chunks -= 1`

`PERCEPTION_RADIUS_M = 60.0`, `CHUNK_SIDE_M = 32.0`.

Soit un agent à la position `(px, py)` dans son home chunk. Pour un chunk à offset `(dx, dy)` (en unités de chunks), la distance minimale d'un point quelconque du home chunk à un point quelconque du chunk-offset est :

```
min_dist = max(0, (|dx| - 1) * CHUNK_SIDE_M)
```

(parce que l'agent peut être sur le bord intérieur de son chunk, donc à distance 0 du bord, et le chunk-offset commence à `|dx|·CHUNK_SIDE_M` puis on retire un côté de chunk pour le tassement).

Pour radius = 60 m, on a besoin d'au moins un chunk avec `min_dist < 60 m`, donc `(|dx| - 1) * 32 < 60` ⟺ `|dx| < 60/32 + 1 = 2.875` ⟺ `|dx| ≤ 2`.

→ `r_chunks = 2` suffit. L'ancien `r_chunks = ceil(60/32) + 1 = 3` était un sur-encadrement par sécurité, qui sortait 24 chunks de plus dans la boucle — tous éliminés par le bbox prefilter de l'optim #2 du sprint précédent. **Pure overhead du for-coord loop, désormais supprimé.**

L'expression `max(1, ceil(radius / CHUNK_SIDE_M))` reste correcte pour les rayons < `CHUNK_SIDE_M` (toujours au moins le home chunk + ses 8 voisins).

## Coût avant / après `_scan_chunk`

Mesuré sur le run 5K de SPRINT-2026-05-15 (`pop=175`, `tick budget = 240 ms`) :

| Étape | Avant (post optim #1+#2) | Coût | Après |
|---|---|---|---|
| Loop `chunks_around(r=3)` | 49 chunks × bbox check | léger | 25 chunks × bbox check |
| Bbox pass-through | ~15 chunks survivent | — | ~15 chunks survivent |
| `_scan_chunk` cell-grid mesh | `XX-px`, `YY-py`, `d2`, `in_range` sur 4096 cellules × 4 ops | ~16K ops / chunk | identique (réutilisé sur les 3 ressources) |
| `(chunk.water > 5).any()` etc. | 3 × 4096 ops + 3 allocations bool array | ~12K ops + 12 KB allocs / chunk | calculé **une seule fois par chunk par tick** (cache) |
| Cells in-range tri | `np.where(mask, d2, inf)` × 3 + `argmin` × 3 | léger (4096 elem) | sur cellules clairsemées (~typique 10-500) |

**Économie théorique** par agent par chunk : ~12K ops + 12 KB d'allocation bool array, fois ~15 chunks/agent × 175 agents = ~31M ops + 31 MB allocs/tick **récupérés**. Sur le frame initial à 295 ms (sprint -3) → 240 ms (post optim #2) → projection ~180-200 ms après cette optim. À vérifier au prochain `profile_tick.py` saturé.

### 3. Déterminisme bit-perfect
A/B avec deux Simulation distinctes, même seed, 50 ticks, hachage SHA-256 de `alive + pos + hunger + thirst` :
```
Run A: 63d0233e83f775c3...
```

Le refactor `np.argmin(np.where(mask, d2, inf))` → `np.argmin(np.where(in_r, d2_sparse, inf))` préserve les tie-breaks parce que `np.nonzero` retourne les indices en ordre row-major (C-order), identique à la flatten 2D originale.

### 4. A/B cache ON / cache OFF
À `pop=20`, le test ne montre pas de gain mesurable (`-0.7 %`) — résultat attendu. La population est trop faible pour que la perception sature, et la majorité des chunks ont 0 cellule de ressource (forêt sans rivière). Le cache est conçu pour briller à `pop ≥ 175` où l'on a 12 chunks scannés × 175 agents = 2 100 appels/tick (et 25M ops mask allocation évitées). À re-mesurer au prochain `profile_tick.py` saturé.
