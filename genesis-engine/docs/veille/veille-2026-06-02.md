## Pistes retenues

### 1. ASAL — Automating the Search for Artificial Life with Foundation Models (Sakana AI)
- **Techno** : framework qui utilise des modèles de fondation vision-langage (CLIP-like)
  pour **chercher automatiquement** des configurations de simulation produisant un
  phénomène cible, de la nouveauté temporellement open-ended, ou une diversité
  « intéressante ». Démontré sur Boids, Lenia, automates cellulaires. Code open source.
- **Apport pour Genesis Engine** : au lieu de régler à la main les paramètres des waves
  (regrow_rate, rayons de perception, coûts), on peut **explorer l'espace des règles** et
  faire émerger automatiquement des régimes vivants/diversifiés — sans script de
  comportement, donc compatible avec l'invariant ZERO PRE-SCRIPT.
- **Module concerné** : observation / recherche de paramètres (`runtime/observation`, harnais
  de smoke), méta-outil hors boucle de simulation.
- **Effort** : M.
- **Risque** : la métrique « intéressant » dépend d'un modèle de fondation externe (coût,
  reproductibilité). À cadrer **hors du chemin déterministe** : l'évaluation FM ne doit
  jamais entrer dans `Simulation.step()`, sinon perte de déterminisme seedé.
- **Sources** : [arXiv 2412.17799](https://arxiv.org/abs/2412.17799) · [page projet](https://pub.sakana.ai/asal/) · [GitHub SakanaAI/asal](https://github.com/sakanaai/asal) · suivi 2025 vision-langage [arXiv 2509.22447](https://arxiv.org/pdf/2509.22447)

### 2. Noyau Rust + PyO3 et ECS Bevy (PyBevy)
- **Techno** : PyO3 (bindings Rust↔Python, référence du domaine) pour pousser les hot loops
  en Rust ; PyBevy expose l'ECS multi-thread de Bevy à Python en s'appuyant sur PyO3, avec
  des systèmes Python tournant **réellement en parallèle** sur plusieurs cœurs (pas de
  sérialisation par le GIL). `cellular_raza` montre un framework de simulation d'agents
  cellulaires bâti en Rust.
- **Apport** : c'est exactement la **Wave 5 (noyau Rust)** du roadmap — montée en échelle
  10⁴–10⁶ agents en extrayant régénération / décision / métabolisme vers `core/` (Rust).
- **Module concerné** : `core/` (futur), pont `runtime/` ↔ Rust.
- **Effort** : L.
- **Risque** : déterminisme cross-langage à garantir (RNG identique, ordre d'itération
  stable, pas de parallélisme non déterministe sur les réductions) ; complexité de build
  (maturin/cibuildwheel) ; supply chain de crates à auditer. **Cyber** : épingler les
  versions de crates, `cargo audit` en CI, pas de `unsafe` non justifié.
- **Sources** : [PyO3 GitHub](https://github.com/PyO3/pyo3) · [pybevy PyPI](https://pypi.org/project/pybevy/) · [PyBevy GitHub](https://github.com/pybevy/pybevy) · [Combining Rust and Python (The New Stack)](https://thenewstack.io/combining-rust-and-python-for-high-performance-ai-systems/)

### 3. Rendu/calcul temps réel WebGPU (compute shaders, MLS-MPM, ping-pong)
- **Techno** : les compute shaders WebGPU (WGSL) permettent des simulations GPU à
  framerate interactif : ~100 000 particules sur GPU intégré (MLS-MPM), fluides, érosion,
  slime mold (Physarum), réaction-diffusion. Architecture ping-pong sur storage buffers.
- **Apport** : double usage — (a) **visualisation temps réel** de 10⁵+ agents pour observer
  l'émergence ; (b) brique de calcul GPU pour les processus environnementaux de la Wave 2
  (hydrologie/érosion déjà amorcées dans le repo, cf. waves 55–56).
- **Module concerné** : rendu/observation (front web), et potentiellement noyau de champs
  environnementaux.
- **Effort** : M (visualisation) → L (si calcul physique porté sur GPU).
- **Risque** : un calcul GPU flottant parallèle est **non déterministe par défaut** →
  réserver le GPU à la **visualisation** et garder la simulation de référence côté
  CPU/Rust déterministe. **Cyber** : valider/sandboxer le WGSL, pas de shader d'origine non
  fiable, CSP stricte côté front.
- **Sources** : [WebGPU fluid sims (Codrops, 2025)](https://tympanus.net/codrops/2025/02/26/webgpu-fluid-simulations-high-performance-real-time-rendering/) · [WebGL vs WebGPU](https://threejsroadmap.com/blog/webgl-vs-webgpu-explained) · [particles-webgpu](https://github.com/piellardj/particles-webgpu)

### 4. Génération procédurale géologie/climat/hydrologie (tectonique + érosion GPU)
- **Techno** : modèles procéduraux fondés sur l'hydrologie (réseaux fluviaux), simulation de
  tectonique des plaques pour montagnes/fosses, érosion hydraulique accélérée GPU, climat
  approximé par cartes de pression au niveau mer. Projets open source de référence.
- **Apport** : alimente le **réalisme Terre** (climat/géologie/hydro) — cohérent avec les
  waves 55–56 (hydrographe, géotherme, faciès métamorphiques) visibles dans l'historique git.
  Sert de **générateur d'environnement initial** crédible sur lequel les agents évoluent.
- **Module concerné** : génération de monde (`worlds/`, `runtime/world`).
- **Effort** : M.
- **Risque** : coût de calcul pour grandes étendues ; veiller à ce que la génération reste
  **seedée et déterministe**. Privilégier les méthodes documentées/reproductibles.
- **Sources** : [Terrain gen via hydrology (ACM TOG)](https://dl.acm.org/doi/10.1145/2461912.2461996) · [realistic-planet-generation (GitHub)](https://github.com/FreezeDriedMangos/realistic-planet-generation-and-simulation) · [Procedural-Tectonics (GitHub)](https://github.com/FioDev/Procedural-Tectonics) · [erosion algo (arXiv 2210.14496)](https://arxiv.org/pdf/2210.14496)

### 5. RL intrinsèquement motivé / open-ended (autotelic, curriculum émergent)
- **Techno** : agents à motivation intrinsèque (nouveauté, progrès de compétence) générant
  et poursuivant **leurs propres buts** (autotelic goal-conditioned RL : CURIOUS, H-GRAIL,
  ExIt 2025). Le curriculum n'est pas prédéfini : il émerge de l'interaction et de signaux
  intrinsèques.
- **Apport** : cadre théorique pour faire émerger des comportements riches **sans script**
  ni récompense imposée — aligné avec l'esprit du projet. Pertinent pour les waves 3–4
  (interactions, proto-culture) comme inspiration de design des besoins/utilités.
- **Module concerné** : `runtime/agents` (modèle de décision/besoins), recherche amont.
- **Effort** : L (et plutôt R&D que livraison directe).
- **Risque** : introduire de l'apprentissage = poids appris ⇒ surveiller déterminisme et
  reproductibilité ; risque de complexité/over-engineering vs. la règle d'utilité actuelle.
  À traiter comme exploration, pas comme dépendance de production.
- **Sources** : [Autotelic agents survey (arXiv 2012.09830)](https://arxiv.org/pdf/2012.09830) · [CURIOUS (arXiv 1810.06284)](https://arxiv.org/pdf/1810.06284) · [Open-ended learning frameworks](https://www.emergentmind.com/topics/open-ended-learning-framework)

## Écartés pour l'instant (notés pour mémoire)
- **World models génératifs (Genie 3 / Project Genie, Waymo World Model)** : impressionnants
  pour générer des mondes jouables photoréalistes, mais **propriétaires, fermés et coûteux**
  (accès Google AI Ultra). Inadaptés à un noyau de simulation déterministe et auditable.
  À surveiller seulement comme couche de rendu/observation expérimentale.
  [Genie 3 (DeepMind)](https://deepmind.google/blog/genie-3-a-new-frontier-for-world-models/) ·
  [Project Genie](https://blog.google/innovation-and-ai/models-and-research/google-deepmind/project-genie/)
- **Agents LLM en boucle (AgentSociety 2)** : architecture inspirante pour simuler des
  sociétés, mais un LLM dans `step()` = **script implicite** (viole ZERO PRE-SCRIPT), casse
  le déterminisme, ajoute coût/latence et **surface d'attaque** (prompt injection, exfiltration).
  Étudier l'architecture, ne pas l'adopter dans le noyau.
  [AgentSociety (arXiv 2502.08691)](https://arxiv.org/abs/2502.08691) ·
  [GitHub](https://github.com/tsinghua-fib-lab/agentsociety)

## Garde-fous cybersécurité (transversal)
Dépendances minimales et **épinglées** (`pip`/`cargo`), `cargo audit` + scan SCA en CI, aucun
`eval`/entrée non validée, modèles et shaders d'origine fiable uniquement, données de
simulation locales (pas d'appel réseau dans le chemin déterministe), revue des crates Rust
tierces avant intégration.
