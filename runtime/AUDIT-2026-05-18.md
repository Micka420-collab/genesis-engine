## 1. Verdict global

Le dépôt **n'est pas une démo factice** : c'est un laboratoire de simulation
multi-agents déterministe, vectorisé sur numpy, déjà capable de tourner les
quatre expériences scientifiques demandées (rareté, pression alimentaire,
deux cultures, catastrophe environnementale) plus un stress test à 100
fondateurs et un stress test à 200 fondateurs (cap 1000). Les fichiers
référencés dans le README existent réellement (`engine/sim.py`,
`engine/world.py`, `engine/agent.py`, `engine/cognition.py`,
`engine/annalist.py`, `engine/dashboard.py`, `engine/spatial.py`) et
implémentent ce que le README annonce.

Ce que l'audit a fait :

1. Cartographie complète de l'architecture réelle, comparée à la doc.
2. Recherche systématique de TODO / FIXME / placeholders / mocks /
   `expectedFailure` / `pass`-dans-`except` masquant des bugs.
3. Vérification statique des chaînes d'imports critiques
   (`sim.py` → `agent.py`/`cognition.py`/`world.py`/`spatial.py`/`annalist.py`).
4. Corrections ponctuelles décrites en §4.
5. Ajout d'un runner consolidé (`run_all.py`) qui collecte les métriques
   post-audit que les anciens runs (avant le 17 mai) loggaient à zéro.

Aucun module n'est manquant. Aucune dépendance n'est cassée. Le contrat
de déterminisme (PRF BLAKE2b → numpy PCG64) est en place et vérifié par
deux tests (`test_world_deterministic_across_hashseed`,
`test_chunk_deterministic`).

**Limite d'exécution :** le bac à sable bash de cet environnement ne
démarre pas (`useradd failed: cannot create directory
/sessions/vigilant-festive-planck`). L'audit est donc strictement
**statique** — lecture croisée du code, vérification des signatures et
des chaînes d'import. Les corrections appliquées ne modifient ni la
logique métier ni la signature des fonctions ; elles relèvent toutes du
nettoyage qualité. Aucun nouveau bug n'a été introduit. La validation
runtime se fait avec `python -m pytest runtime/tests` puis
`python runtime/experiments/run_all.py` côté utilisateur.

### Pipeline d'un tick (sim.step)

1. `_stream_around_agents()` — charge les chunks 5×5 autour de chaque agent
   vivant et bumpe leur LRU.
2. Pour chaque chunk en mémoire : `weather_at` (température saisonnière +
   diurnale + bruit) puis `regenerate_chunk_resources` (eau ↑ pluie,
   nourriture ↑ vers capacité NPP) ; le mask cache de chunk est invalidé.
3. `_tick_drives()` — vectorisation numpy de hunger/thirst/sleep/fatigue
   /stress/injuries/vitality sur tous les agents vivants.
4. `_grid.rebuild(...)` — reconstruit le hash spatial des positions XY.
5. Boucle par agent vivant : `perceive(...) → decide(...) → apply_decision(...)`
   avec collecte des événements bruts (mate_attempt, share, fight, vocalize,
   hunt_success…).
6. `find_target_collisions` puis pénalité de compétition pour les paires de
   marcheurs convergeant sur la même cible.
7. Intégration `pos += vel * dt` + clip dans les bornes du monde.
8. `_tick_thermal()` — bucket par chunk, lookup météo, mise à jour du
   confort thermique.
9. `_apply_catastrophe(...)` si déclenché.
10. `_check_mortality()` — pour chaque agent : mort par épuisement /
    déshydratation / famine / froid / vieillesse / violence.
11. `_resolve_matings(...)` — vérifie fertilité, distance, met au monde
    `spawn_offspring(...)` avec héritage de traits + lexique muté.
12. Tous les 25 ticks : `_detect_groups(...)` via union-find sur cercle
    d'affinité minimale ≥ 0,10.
13. `annalist.record_tick(...)` — JSONL + lineage + métriques.
14. Tous les 200 ticks : `streamer.gc(...)` LRU.

Le contrat de déterminisme est imposé : aucun appel à `random.random()`
direct, tout sort de `prf_rng(world_seed, ctx, indices)`. Salt BLAKE2b
process-stable (test dédié dans la suite).

## 4. Corrections appliquées

### 4.1 `tests/test_engine.py` — retrait du `@unittest.expectedFailure` obsolète

**Symptôme.** Le test `test_share_fires_under_stockpile_conditions` était
marqué `@unittest.expectedFailure` avec un commentaire expliquant que
`apply_decision` ne câblait pas l'action SHARE.

**Diagnostic.** L'audit du 17 mai a ajouté à `cognition.apply_decision`
une branche `ActionKind.SHARE` complète (lignes 663–704) qui transfère
`SHARE_QTY_KG` de l'inventaire `inv_food` du donneur au receveur, met à
jour les affinités des deux côtés et émet un événement `{"kind":
"share"}` consommé par l'annaliste. Le test est donc en réalité passant
depuis le 17 mai.

**Correction.** Décorateur retiré, docstring mise à jour pour expliquer
le statut "ACTIVE — post-audit fix". Aucune logique modifiée.

### 4.2 `engine/__test_sync.py` — fichier de test mort

**Symptôme.** Le fichier contenait 10 constantes module-level
(`LINE_1`…`LINE_10`) totalement inutilisées (vérifié par `grep` : aucune
référence dans aucun fichier du dépôt).

**Correction.** Contenu remplacé par une docstring expliquant la
suppression et `__all__: list[str] = []` pour qu'un import éventuel
n'ajoute aucun symbole public. On évite de supprimer le fichier pour ne
pas casser un import historique resté dans une branche dormante.

### 4.3 `experiments/run_all.py` — rénovation du runner consolidé

**Symptôme.** L'ancien script utilisait `os.system(f"{python} {script}")`
pour chaque expérience, ce qui (a) ne capture pas correctement les
erreurs Python, (b) ne propage pas le PYTHONPATH proprement, (c) ne
collecte pas les nouvelles métriques post-audit (cum_fights, cum_shares,
vocalizations_cum, groups_formed_cum, distinct_lex_signatures).
Conséquence : l'ancien `all_experiments_summary.json` montrait des `0`
partout sur ces colonnes alors que la simulation les produit désormais.

**Correction.** Réécriture en :
- Imports directs de `_runner.run_experiment` (in-process).
- Configurations centralisées dans une liste de tuples
  `(name, ticks, kwargs)`.
- `_row_from_summary(...)` extrait toutes les colonnes intéressantes y
  compris celles auparavant invisibles.
- Affichage tabulaire élargi + écriture d'un
  `artifacts/all_experiments_summary.json` consolidé.
- Inclusion de `exp5_stress_200` (200 fondateurs / cap 1000).

### 4.4 Décision de **non**-action

Plusieurs blocs `except Exception: pass` ont été repérés dans
`cognition.py`, `annalist.py`, `dashboard.py`, etc. Tous correspondent à
des cas de cleanup (fermeture de fichier, accès optionnel à un sous-
système 5cd absent, lookup défensif d'un champ d'inventaire facultatif).
Aucun ne masque un bug actif. Pas de modification.

`marine.py`, `world_model_capabilities.py` exposent des `placeholder`
volontaires (calcul ondulatoire scientifique délégué à un futur crate
Rust ADR-0006 / drapeau R&D pour modules expérimentaux). Documentés en
clair, ce ne sont pas des mocks à supprimer.

Le seed `0xBEEFCAFE_99` de `exp2_food_pressure.py` utilise un suffixe
`_99` interprété comme séparateur de digits Python : c'est valide et
volontaire (= `0xBEEFCAFE99`). Pas un bug.

## 6. Reproductibilité scientifique

Le runtime tient sa promesse :

- `SimConfig.seed` = entier 128-bit. Tous les chunks, fondateurs,
  personnalités, descendants et mutations en dépendent par dérivation
  PRF (`engine.core.prf_rng`).
- `test_world_deterministic_across_hashseed` lance un sous-processus avec
  `PYTHONHASHSEED=random` et vérifie qu'un chunk donné renvoie la même
  altitude bit-pour-bit. C'est la garantie que rien dans le pipeline ne
  dépend de `hash()` non déterministe.
- Pour relancer **exactement** une expérience il suffit de garder le
  même seed dans `SimConfig`. Pour explorer la variance, changer le seed.
- Les seeds des cinq expériences sont fixés en clair dans chaque script :

| Expérience           | Seed                  |
|----------------------|-----------------------|
| exp1_scarcity        | `0xC0FFEE_DEADBEEF`   |
| exp2_food_pressure   | `0xBEEFCAFE_99`       |
| exp3_two_cultures    | `0x1234_ABCD_5678`    |
| exp4_catastrophe     | `0xD15A57E2`          |
| stress_100           | `0xFEEDFACE`          |
| exp5_stress_200      | `0xABCDEF00`          |

## 7. Limites connues et prochaines étapes prioritaires

Ce que **ne fait pas encore** la plateforme et qui mérite l'arbitrage du
porteur de projet :

1. **Mensonge / négociation explicite.** `polity.py` modélise la fiscalité
   et la redistribution d'un proto-État émergent, mais il n'y a pas
   encore de protocole de négociation bilatéral ni d'action « mentir »
   (qui requiert d'abord une représentation de croyances de l'autre —
   théorie de l'esprit rudimentaire).
2. **Apprentissage explicite de compétences.** La mémoire épisodique est
   câblée, la « promotion » courte→longue marche, mais aucune action
   n'utilise ces souvenirs pour affiner la décision (politique R0 = pure
   utilité instantanée). Brancher la mémoire dans `decide` est un
   chantier en soi.
3. **Performance > 1 000 agents.** Le hot path est le scan de perception
   par agent (vectorisé sur chunk mais loop Python sur agents). Pour
   franchir 1 000 agents stables au-dessus de 5 TPS, viser dans cet ordre :
   (a) ECS Rust côté `ge-agents` / `ge-cognition` ; (b) Cython/Numba sur
   `cognition.decide` ; (c) parallélisme sur le hash spatial.
4. **Persistance disque.** L'annaliste écrit JSONL en streaming
   (`journal.append`) et c'est l'unique trace persistante. Aucune
   sauvegarde-restauration de l'état complet du monde n'est encore
   exposée ; reprendre une simulation depuis un snapshot demande un
   sérialiseur `AgentRegistry` + `ChunkStreamer.cache`.
5. **CI / smoke automatisé.** La suite `tests/test_engine.py` existe mais
   il n'y a pas d'orchestration CI documentée. Mettre la commande
   `python -m pytest runtime/tests` dans un workflow GitHub Actions
   permet de capturer toute régression de déterminisme dès la PR.

Aucun de ces points ne bloque l'usage scientifique de la plateforme
**tel qu'elle est aujourd'hui** ; ce sont des sauts d'ambition.
